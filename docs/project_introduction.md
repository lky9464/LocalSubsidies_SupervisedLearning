# 지방보조금 부정수급 위험도 측정 시스템
## 프로젝트 소개 (로컬 웹 서비스)

**버전:** 2026-07  
**대상:** 업무 담당자·관리자 (비전문가 포함)  
**원칙:** 모든 데이터·학습·점수는 **사용자 PC 내부(127.0.0.1)** 에서만 처리됩니다.

---

## 1. 프로젝트 개발 목적

### 1.1 배경

지방보조금 사업은 건수가 많고, 부정수급·오지급 사례는 전체 대비 **매우 적은 비율**로 발생합니다.  
사람이 모든 건을 동일하게 점검하기는 어렵고, **위험도가 높은 건을 우선** 살펴볼 기준이 필요합니다.

### 1.2 본 시스템이 하는 일

과거 데이터(적발·등록·확정 등 **타겟 정보가 있는 구간**)로 **지도학습 모델**을 만든 뒤,

- 각 사업 건에 **위험도 점수(0~1,000)** 를 부여하고
- **주 모델·보조 모델** 점수를 조합해 **점검 우선순위**를 정리하며
- **라벨이 아직 없는 최신 데이터(추론)** 에도 같은 방식으로 점수를 적용합니다.

### 1.3 기대 효과

| 구분 | 효과 |
|------|------|
| 점검 효율 | 한정된 인력으로 **고위험 건 우선** 점검 |
| 객관성 | 동일 규칙·동일 모델로 **재현 가능한** 우선순위 |
| 보안 | raw·점수·모델을 **외부 클라우드 없이** 로컬 PC에서만 처리 |
| 검증 | 평가(Test) 구간에서 **타겟 포착 품질**을 먼저 확인 후 추론에 적용 |

### 1.4 사용하는 학습 방법(알고리즘) — 개요

5종 알고리즘을 **학습·비교**합니다. **주 모델·보조 모델**은 고정이 아니라,  
평가(07)·순위(08) 결과와 `configs/default.yaml`의 `ops_queue` 설정으로 정합니다.

| 알고리즘 | 특징 (일반) |
|----------|-------------|
| **RandomForest** | 다수결 트리 — 안정적 baseline, 주·보 후보로 자주 1~2위 |
| **CatBoost** | 범주형 강한 부스팅 — 교차 확인·보조 후보 |
| **Stacked / EasyEnsemble / Gradient Boosting** | 비교·참고 — 순위에 따라 주·보가 될 수 있음 |

**현재 스냅샷(2026-07):** RandomForest(주) + CatBoost(보) — [`operations_criteria.md`](operations_criteria.md) 참고. 재학습·데이터 변경 시 **바뀔 수 있음**.

## 2. 환경 구성 · 요구사항 · 주의사항

### 2.1 전체 구성(폴더)

```
[형제 폴더 권장]

LocalSubsidies_SupervisedLearning/     ← 프로그램·웹 UI (GitHub 가능)
  ├── api/          FastAPI
  ├── web/          Next.js (빌드 → web/out, Release로 배포)
  ├── scripts/      파이프라인
  ├── configs/      설정 (local.yaml은 PC마다 다름)
  └── RunWebNext.bat 웹 서비스 실행 (:8600)

LocalSubsidies_ML_Data/                ← 데이터 전용 (프로젝트 밖·GitHub 금지)
  ├── raw/              학습·평가용 원본 CSV
  ├── raw_inference/    추론용 원본 CSV
  ├── interim/          통합·라벨 중간 파일
  ├── processed/        전처리·분할 결과
  ├── algorithms/       모델·점수 (알고리즘별)
  ├── ops/              운영 DB (ops.sqlite, raw 미포함)
  └── runs/             실행 로그·Job 기록
```

**핵심:** 민감한 CSV·행단위 점수·모델은 **항상 `data_root`(ML_Data) 쪽**에만 둡니다.

### 2.2 PC 요구사항 (사양에 따라 작업 능력이 달라짐)

웹 메뉴 **「내 PC 사양 체크」** 로 본인 PC를 확인할 수 있습니다.

| 항목 | 쾌적 | 보통 | 부족 | 업무 영향 |
|------|------|------|------|-----------|
| RAM | ≥32GB | 16~32GB | <16GB | 학습(05) 메모리 부족·중단 가능 |
| CPU 논리코어 | ≥8 | 4~7 | <4 | 학습·전처리 **수 시간~반나절** 소요 |
| 디스크 여유 | ≥50GB | 20~50GB | <20GB | 모델·점수 파일 저장 공간 부족 |
| OS | — | Windows 10/11 64bit | — | macOS/Linux는 스크립트 수정 필요 |

**같은 버튼을 눌러도 PC마다 걸리는 시간이 다릅니다.**

- **쾌적:** 5종 알고리즘 일괄 학습·평가 가능
- **보통:** 가능하나 **수 시간** · 절전 모드 OFF 권장
- **부족:** 알고리즘 **2~3개만** 선택 · 일괄 실행 지양

개발·검증에 사용된 참고 사양: Ryzen 3 3200G, RAM 약 14GB — **보통~부족 경계** 수준입니다.

### 2.3 raw 데이터 관리 방안 (필독)

#### 왜 중요한가

raw CSV에는 **사업·기관 정보**와 **식별에 해당할 수 있는 컬럼**이 포함될 수 있습니다.  
유출 시 개인정보·업무 기밀 사고로 이어질 수 있으므로 **별도 관리**가 필수입니다.

#### 권장 원칙

| 원칙 | 설명 |
|------|------|
| **위치 분리** | raw는 `LocalSubsidies_ML_Data` 등 **프로젝트 밖** 전용 폴더 |
| **GitHub 금지** | raw·점수·모델·ops.sqlite **커밋·업로드 금지** |
| **AI 도구 주의** | Cursor 등에 CSV 내용·샘플 행 **붙여넣기 금지** |
| **웹은 로컬만** | 주소는 **127.0.0.1** 만 사용 (`0.0.0.0`·터널·클라우드 배포 금지) |
| **DB에는 메타만** | 웹 DB에는 파일명·건수·해시 등 **메타만**, raw 내용 미저장 |
| **백업·접근 통제** | USB·공유폴더·동기화 클라우드에 raw 올릴 때 **암호화·권한** 검토 |

#### 학습용 vs 추론용 raw

| 폴더 | 용도 | 타겟(TAET_YN 등) |
|------|------|------------------|
| `raw/` | 모델 **학습·평가** | 있음(또는 라벨 수정용 컬럼) |
| `raw_inference/` | **추론**(미지 데이터) | 없어도 됨 — 시스템이 무시 |

스키마는 `TLS4902R_Layout` 기준으로 **동일 레이아웃**입니다.

### 2.4 소프트웨어 설치 (최초 1회)

1. **Python 3.10+** 설치 (Windows 64bit)
2. 프로젝트 폴더에서:
   ```
   python -m venv .venv
   .venv\Scripts\activate
   pip install -r requirements.txt
   ```
3. 설정 파일:
   ```
   copy configs\local.yaml.example configs\local.yaml
   ```
   `data_root`를 본인 PC의 `LocalSubsidies_ML_Data` 경로로 수정
4. `data_root` 아래 `raw/`, `raw_inference/` 폴더 생성

### 2.5 주의사항 요약

- 브라우저 주소는 **http://127.0.0.1:8600**
- **RunWebNext.bat 검은 창을 닫으면** 웹·학습 Job이 **함께 종료**됩니다
- 코드·UI 수정 후에는 **build_web.bat**(필요 시) 후 **RunWebNext.bat / RestartWeb.bat 재시작**
- 학습 Job은 **백그라운드** — 다른 메뉴로 이동해도 진행 (상단 배너 확인)
- **과금되는 외부 API·클라우드 없음** — 전기·본인 PC만 사용

## 4. 전반 업무 순서 (웹 기준 · 모든 케이스)

### 4.1 큰 그림

```
[준비] PC 사양 확인 → data_root·raw 배치 → RunWebNext.bat 실행
                    ↓
        ┌───────────────────────────────────┐
        │  A. 최초·정기: 모델 학습 및 평가      │
        │     (라벨 있는 raw)                  │
        └───────────────────────────────────┘
                    ↓
        ┌───────────────────────────────────┐
        │  B. 운영: 추론 (라벨 없는 raw)       │
        │     학습된 모델로 최신 건 점수 산출    │
        └───────────────────────────────────┘
```

### 4.2 A. 학습·평가 — 정상 흐름

```
데이터 등록(raw 업로드)
    → 학습 파이프라인
        01 원본통합 → 02 타겟수정 → 03 전처리
        → 04 누수점검 ──PASS──→ 05 학습 (2종 이상 선택)
        → 06 Feature중요도 → 07 평가·점수
        → 08 모델순위 → 09 리포트 → 10 타겟 포착 분포
    → 모델 비교·평가 (지표·차트 확인)
    → 타겟 포착 분포 (Test: 포착 품질 확인)
    → 대시보드 요약
```

### 4.3 A-예외. 04 누수점검 FAIL

```
04 FAIL → 의심 피처 목록 확인
    → 제외할 피처 선택
    → 「제외 반영 후 03부터 재개」
    → 03 전처리 → 04 재실행
    → PASS 후 05부터 계속
```

**누수:** 모델이 “정답을 미리 아는” 컬럼을 학습하면 점수가 비현실적으로 좋아집니다. 반드시 PASS 후 학습.

### 4.4 A-예외. 05 학습 중단·재실행

```
Job 실패 또는 PC 재부팅
    → RunWebNext.bat 재실행
    → 실패 단계부터 다시 실행 (또는 05만 재실행)
    → 상단 배너·Run 이력에서 상태 확인
```

### 4.5 B. 추론 — 운영 흐름

```
(선행) A 경로에서 05·06까지 완료된 모델 존재
    → 데이터 등록(raw_inference 업로드)
    → 추론 → 실행 (주·보 모델 2종 이상 — `ops_queue` 설정과 동일 알고리즘)
    → 추론 → 결과 확인
        · 점검 우선순위표(4×4) 미리보기
        · Excel 생성 → ops_queue_inference.xlsx
    → 대시보드 추론 요약 확인
```

**6월 데이터가 나중에 들어오는 경우:** CSV 추가 후 **11 추론 재실행** (1~6월 통합 재산출).

### 4.6 케이스별 “어디서 멈추고 무엇을 보나”

| 상황 | 다음 행동 | 확인 메뉴 |

## 6. 학습·평가 vs 추론 — 차이와 업무 활용

### 6.1 한 줄 비교

| | **학습·평가** | **추론** |
|---|-------------|---------|
| **데이터** | 과거·**타겟(적발 등) 알려진** raw | **타겟 미지** raw_inference |
| **목적** | 모델 만들기 · **얼마나 잘 맞는지 검증** | **앞으로 점검할 건** 고르기 |
| **웹 경로** | 모델 학습 및 평가 | 추론 |
| **대표 산출** | 모델 파일 · Test 점수 · 타겟 포착 분포 | inference 점수 · **점검 우선순위표** |
| **실제라벨** | 있음 → 성능·포착률 계산 가능 | 없음 → “맞췄는지”는 나중에 검증 |

### 6.2 점검 우선순위 규칙 (공통 개념)

주·보 모델 각각 점수 **상위%** 로 A~D 구간을 나눕니다.  
(어떤 알고리즘이 주·보인지는 **설정·순위**에 따름 — 고정 아님)

| 구간 | 의미 |
|------|------|
| **A** | 상위 1% |
| **B** | 1% 초과 ~ 5% |
| **C** | 5% 초과 ~ 10% |
| **D** | 10% 초과 |

**4×4 우선순위(1~16):** 주등급이 먼저, 같은 주등급 안에서 보조등급 순.  
**작을수록** 점검 우선.

### 6.3 업무 활용 예시

#### 예시 1 — 연 1회 모델 갱신 (학습·평가)

> 2024~2025년 적발·등록 데이터로 모델을 학습하고, 2025년 하반기 Test로 성능을 확인한다.

1. `raw/`에 학습용 CSV 배치  
2. 파이프라인 01~10 실행  
3. **모델 비교·평가**에서 **1~2위(주·보 후보)** 순위 확인 · `ops_queue` 설정 점검  
4. **타겟 포착 분포**에서 “상위 1%에 실제 타겟이 얼마나 모였는지” 확인  
5. PASS·성능 만족 시 → **확정된 주·보 모델**로 추론에 사용

#### 예시 2 — 분기별 선제 점검 (추론)

> 2026년 1~5월 데이터는 아직 적발 결과가 없다. 위험도가 높은 사업부터 점검 계획을 세운다.

1. `raw_inference/`에 202601~202605 CSV  
2. **추론 실행** (대시보드·설정에 표시된 **주·보** 알고리즘 — 보통 2종)  
3. **결과 확인** → 우선순위 1~5위 Excel 추출  
4. 감사·점검팀 회의 자료로 **우선 점검 명단** 활용  
5. 6월 CSV 확보 후 → 추론 **재실행**해 명단 갱신

#### 예시 3 — 주·보조 불일치 건 처리

> 주 모델은 A구간인데 보조는 D구간인 건이 있다.

- **의미:** 한 모델만 고위험 — **과탐 또는 애매한 건** 가능  
- **업무:** 무조건 최우선 점검보다 **사유 확인·경미 점검** 후 결정  
- 반대로 **주A×보A(우선순위 1)** 는 **최우선 점검** 후보

#### 예시 4 — PC가 느린 팀

> RAM 8GB PC에서 5종 전체 학습이 실패한다.

1. **내 PC 사양 체크** → 부족 확인  
2. 파이프라인에서 **주·보 2종만** 선택해 05 실행 (대시보드에 표시된 알고리즘)  
3. 추론도 동일 2종 — **4×4·우선순위**에 필요한 최소 구성
